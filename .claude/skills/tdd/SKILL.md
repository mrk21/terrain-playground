# tdd

RED → GREEN → REFACTOR のサイクルで、テストを先に書きながらロジックを実装する。各フェーズの境目で必ず検証コマンドを回し、「正しい状態」を確認してから次へ進む。

## 対象範囲

テスト可否を分けるのは「純粋関数か」ではなく「**実行環境（ブラウザ/WebGL/DOM）に依存するか**」。vitest は `environment: node` なので、素の値（数値・文字列・プレーンなオブジェクト）と標準ライブラリだけで動くコードはそのままテストできる。`HTMLCanvasElement`・`WebGL2RenderingContext`・`document`・`PointerEvent`・`DOMRect` などブラウザ固有の API・型に触れるコードは、jsdom 等のモックを入れない限り node では動かない（純粋関数でも、ブラウザ型を使えばモックが要る）。

- **TDD する**: `algorithm/`・`core/math`・`core/colormap`。数値・配列・プレーンなオブジェクトだけで完結し環境に依存しないので、node でそのままテストできる。
- **そのままでは TDD しない**: `visualization/`（`gl/`・`scenes/`・`ui/`・`input/`）の WebGL/DOM/canvas に直接触るグルー。実行環境（ブラウザ）が要るのでユニットテスト向きでない。
- **抽出できるなら抽出する（ただし程度問題）**: その中に検証する価値のある計算（クランプ・補間・座標変換・状態遷移など）が埋まっていたら、環境に依存しない関数として切り出して TDD する（行き先: 汎用→`core` / 画面・入力固有→`visualization/` 内ヘルパ。`algorithm/` は不可）。一方で**テストのために設計を歪めない** — 不自然な型変換や薄い間接層が増えるだけなら、無理に出さずグルーのまま対象外にするか、その箇所だけ jsdom 等でモックする。判断材料は reference.md の「テスト可否を分ける軸」「レイヤと抽出方針」。

## サイクル

### RED — 失敗するテストを先に書く

これから足す/直す振る舞いを表すテストを書く。**実装コードはまだ書かない**。

正常系だけでなく、**退化・境界の入力（ゼロ・空・0 除算・範囲端など）もここで洗い出してテストに含める**。GREEN は「テストを通す最小」なので、RED で書かなかったケースは実装されない（例: ベクトル正規化ならゼロベクトル、補間なら t=0/1）。逆に、起きえない入力のために投機的な分岐を増やす必要はない。

`npm run test:run` を回し、**そのテストが、まだ実装が無いという正しい理由で失敗している**ことを確認する。落ちない・別の理由（import エラー等）で落ちる場合はテスト側を直す。失敗を確認できて初めて次へ進む。

### GREEN — 最小実装で通す

そのテストを通すだけの**最小限**の実装を書く。先回りして使われていない分岐や汎用化を足さない（それは次の機能の RED で driving する）。

`npm run test:run` を回し、**全テストが緑**になることを確認する。

### REFACTOR — 緑のまま整理する

テストを緑に保ったまま、命名・重複・見通しを整える。リファクタ中も小まめに `npm run test:run` で緑を保つ。

仕上げに `npm run check`（Biome の lint+format、自動修正される）と `npx tsc --noEmit`（型チェック）を回し、**警告・エラーが残っていない**ことを確認する。複雑度の警告（`noExcessiveCognitiveComplexity`）が出たときの扱いは CLAUDE.md「複雑度と設計見直し」に従う（既定は再設計、本質的な複雑さのみ理由付き `biome-ignore`）。

## 完了ゲート

次の 3 つが揃って初めて「完了」とする:

1. `npm run test:run` が全緑
2. `npx tsc --noEmit` が型エラー 0
3. `npm run check` がクリーン（複雑度の警告も、解消するか理由付きの `biome-ignore` で残さない）

このプロジェクトの具体的なコマンド・テストの書き方（vitest / 日本語テスト名 / `*.test.ts` の置き場所）・抽出の指針は [reference.md](reference.md) にまとめてある。サイクルに入る前に一読する。
