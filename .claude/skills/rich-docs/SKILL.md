# rich-docs

`docs/*.md` の図と数式を、GitHub と VS Code が**ネイティブに描画できる**表現へ引き上げ、`npm run docs:render`（`tools/render-doc.mjs`）で**実際に描かれた絵を見て**正しさを確かめる。アプリの絵を守る visual-check の「ドキュメント版」にあたり、役割分担している——**アプリ/WebGL の描画は visual-check、ドキュメントの図・数式はこのスキル**。

中心にある考え方はひとつ:

> 図は種類で道具を変える。Mermaid を万能視せず、**関係＝Mermaid / 数式＝LaTeX / 幾何＝SVG** に振り分ける。そして「置換した」で終わらせず、**焼いて目で見る**まででワンセット。

## いつ使うか

- **使う**: `docs/*.md` に図・数式を**足す/直す**とき。ASCII アートの図、``` フェンスにベタ書きした数式、テキストだけで説明している関係・フロー・木構造を見つけたら、頼まれなくても3層化を提案・実施する。新しい設計メモを `docs/` に書くときも最初から3層で。
- **使わない**: アプリ/WebGL の描画（シーン・シェーダ・カメラ）→ `visual-check`。`algorithm/`・`core/` の純粋ロジック → `tdd`。README の実体同期 → `update-readme`。`docs/` 以外の Markdown（README 等）の図はこのスキルの対象外（必要なら同じ3層の考え方は流用できる）。

なぜ3層に分けるか: Mermaid は**自動レイアウトの関係図専用**で、座標を自分で置く幾何図（格子・断面・横から見た図）は表現できない。逆に数式や自由な図を Mermaid に押し込むと崩れる。各表現を得意な道具に任せるのが、結局いちばん速くて崩れない。

## 図を3層に振り分ける

| 図の種類 | 道具 | 置き方 |
| --- | --- | --- |
| 関係・フロー・木構造（判断ツリー、1フレームの流れ、親子関係） | **Mermaid** | ` ```mermaid ` フェンス（`flowchart TD` など） |
| 数式（導出、比、log/tan、定義） | **LaTeX** | インライン `$...$` / ブロック `$$...$$` |
| 幾何・空間（格子の分割、視錐台、断面、座標つきの図） | **SVG ファイル** | `docs/assets/*.svg` を `![説明](assets/x.svg)` で埋込 |

**そのまま残すもの**（変換しない）: 実 TS コード片・定数定義ブロック・既に読みやすい Markdown テーブル。コードはコードのまま ``` で。**「数値を割るための機械的変換」はしない**——意味のある図・数式だけを引き上げる。

**「リッチ化して」では図を取りこぼさない。** 「分かりやすくして」「リッチに」と言われたら、そのドキュメントにある ASCII 図・ベタ書き数式は**ひととおり全部**3層化する。ユーザーが「式とフローだけ」のように一部を名指ししても、それは "最低限ここは" の意味に取り、他に明らかに見劣りする ASCII 図が残るなら同じ品質まで引き上げる。磨いた Mermaid/LaTeX の隣に生の ASCII 図を1枚残すと、そこだけ浮いて「結局よく分からない」と言われる——実際のレビューで出た不満。本当に1か所だけにしたいときはユーザーがそう言う。

判断に迷う図の具体例と、各道具の書き方の要点・落とし穴・配色は **[reference.md](reference.md)** にまとめてある。Mermaid の構文エラーで詰まったとき、SVG の配色を決めるとき、KaTeX のエスケープで悩んだときに読む。

## ワークフロー

### 1. 置換する（3層化）

対象 .md を読み、上の表で図を振り分けて置換する。要点だけ:

- **Mermaid**: ラベルに `/` `?` などを含むなら `"..."` で囲む。`>` は `&gt;`、改行は `<br/>`。詳細は reference.md。
- **LaTeX**: **数式に `_`（アンダースコア）を入れない**——GitHub は `\text{}` 内の下線で数式ごと壊れる（reference.md 参照）。`SCREAMING_SNAKE` 定数は数式に直書きせず**短い記号＋凡例**に（式では $B$、本文で「$B$ は `BASE_TILE_WORLD`」）。定数名はコードスパン \`...\` なら下線も安全。ブロックは `$$ ... $$`（複数行可）。`\log_2` の math mode 添字 `_{...}` は問題ない。
- **SVG**: **インライン `<svg>` ではなくファイル参照**（GitHub のサニタイザに左右されないため）。`docs/assets/` に置く。**light/dark 両テーマで潰れない配色**にする（中間色ストローク・塗りは none/半透明・純黒純白を避ける。パレットは reference.md）。`viewBox` はテキストがはみ出さない大きさに。
  - **ASCII の丸写しにせず、概念の肝を絵にする。** 元図が「層の重ね合わせ・累積・before→after」のような**合成**を表しているなら、部品を並べるだけでなく**合成した結果**まで描く。例: fBm のオクターブ図なら、各オクターブの波だけでなく "足し合わさった最終形" の波も1本描くと「どう合成されるか」が一目で伝わる。読者が知りたいのは部品ではなく**組み合わさり方**——ここを省くと「合成している感がない／よく分からない」になる（実際のレビューで最も効いた差）。

### 2. 焼いて目で見る（docs:render で検証）

置換しただけでは「Mermaid が構文エラーで出ない」「数式が崩れる」「SVG が暗背景で潰れる」「テキストが viewBox から切れる」に気づけない。**必ず PNG に焼いて Read で確認する**。

```sh
# 全体を light/dark 両方で焼く（test-results/docs/ に出力。gitignore 済み）
npm run docs:render -- docs/<file>.md --theme both

# 図ごとに等倍で抜き出してズーム確認（細部の検証に有効）
npm run docs:render -- docs/<file>.md --clip .mermaid          # Mermaid 図ごと
npm run docs:render -- docs/<file>.md --clip .katex-display    # 数式ブロックごと
npm run docs:render -- docs/<file>.md --theme dark --clip "article img"  # SVG をダークで
```

確認の観点（**画像を Read して目で見る**。出力に `⚠ mermaid:` が出ていないかも見る）:

- **Mermaid**: 図が描かれているか（構文エラーだと出ない／壊れる）。ラベルの `>` や改行が意図通りか。
- **LaTeX**: 分数・上付き下付き・log/tan・矢印が正しく組版されているか。**数式に `_` を入れていないか**（`\text{}` 内の下線は GitHub で壊れる。docs:render の KaTeX は通してしまうので別途チェック）。
- **SVG**: **light と dark の両方**で線・文字・塗りが読めるか。`viewBox` から文字がはみ出て切れていないか。
- 既存の SVG を直したら、Quick Look でも一発確認できる: `qlmanage -t -s 900 -o /tmp docs/assets/x.svg`。

### 3. 直して再確認

崩れ・切れ・潰れがあれば直して 2 に戻る。SVG のはみ出しは `viewBox` を広げるかラベルを分割／短縮。Mermaid のエラーはラベルの引用・エスケープを見直す。**意図どおり描けた画像を1枚は実際に見るまで完了にしない。**

## 注意

- `test-results/docs/` は使い捨て（`.gitignore` 済み）。コミット対象は `docs/*.md` と `docs/assets/*.svg`。
- npm スクリプトやディレクトリを増減したら README 同期（`update-readme`）も忘れずに。`docs:render` 自体は導入済み。
- VS Code の組込みプレビューは LaTeX・SVG は出るが **Mermaid だけ拡張が必要**。最終の正本確認は GitHub 上（または docs:render の PNG）で。

## 完了ゲート

- 3層化した各ブロックが **docs:render の PNG で正しく描画**されることを目で確認した（Mermaid・LaTeX・SVG）。
- SVG は **light/dark 両テーマ**で読め、`viewBox` 内に収まっている。
- `⚠ mermaid:` 警告が出ていない。
- 実コード片・定数・テーブルは触らず残してある（機械的変換をしていない）。
