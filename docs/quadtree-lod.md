# クアッドツリー LOD（Quadtree Level of Detail）まとめ

地形を「ズームしても描画コストがほぼ一定」「拡大しても粗くならない」で描くための仕組み。
このリポジトリでは **2D（俯瞰）** と **3D（地形メッシュ）** の両方で使っているが、
中身はだいぶ違う。まず共通の考え方を押さえてから、2つの実装を見ていく。

- 2D: [src/visualization/scenes/scene-heightmap.ts](../src/visualization/scenes/scene-heightmap.ts)（タイルピラミッド）
- 3D: [src/visualization/scenes/scene-heightmap3d.ts](../src/visualization/scenes/scene-heightmap3d.ts)（適応的四分木）

---

## 0. なぜ LOD が要るのか

地形は理屈の上では「無限に細かい」。高さ関数 `height(x, z)` はワールド座標の**どんな細かさ**でも
値を返せる（[height.ts](../src/algorithm/height.ts)）。これをそのまま描こうとすると 2 つの問題が出る。

- **広く引いて見るとき**: 画面に映るワールドが広大なので、1 ピクセルに何百テクセル／三角形分もの
  地形が詰まる。全部を最大解像度で作ると、見えもしない細かさのために大量のメモリと計算を使う。
- **拡大して見るとき**: 固定解像度の 1 枚絵を引き伸ばすと、拡大コピーのように**ガビガビ**になる。

LOD（Level of Detail）の発想は単純で、

> **画面に映る粗さ（1 ピクセルあたりの細かさ）に見合った解像度だけを、その都度作る。**

引いたら粗く、寄ったら細かく。しかも「寄った」ときは固定絵を拡大するのではなく、
`height()` を**その細かさで取り直す**。だから無限にズームしても常にくっきりする。
これを実現する空間分割が **四分木（quadtree）** ＝ 正方形を 2×2 に再帰分割する木。

```
        ┌───────────────┐         ┌───────┬───────┐         ┌───┬───┬───────┐
        │               │         │       │       │         │   │   │       │
        │   depth 0     │  分割   ├───────┼───────┤  さらに ├───┼───┤ d1    │
        │   (1 ノード)  │  ────►  │       │       │  分割   ├───┼───┤       │
        │               │         │       │       │         │d2 │d2 │       │
        └───────────────┘         └───────┴───────┘         └───┴───┴───────┘
        ルート = 広く粗く          1 段で 4 ノード            細かい所だけ深く
```

レベル（深さ）が 1 上がるごとに一辺は半分、面積は 1/4、同じ面積を覆うノード数は 4 倍になる。

## 1. 共通の骨格（2D / 3D で同じ考え方）

実装は別物だが、次の 4 つは両方に共通する。

### 1-1. ノードはキー付きでキャッシュする

ノード（2D ではタイル、3D ではメッシュ）は `(レベル, セル座標)` をキーに `Map` で持つ。

| | キーの形 | レベルの呼び方 |
| --- | --- | --- |
| 2D | `` `${level},${cx},${cz}` `` | level |
| 3D | `` `${depth}:${nx}:${nz}` `` | depth |

レベル `L` のセル座標 `(c)` が指すワールド原点は `c * (基準サイズ / 2^L)`。
セル座標は整数なので、**同じレベルのタイルは必ず格子に整列**し、隣と継ぎ目なく並ぶ。

### 1-2. 毎フレーム「使った印」を付け、使わなかったものを捨てる

`frame` というカウンタを毎フレーム 1 増やす。描画・参照したノードに `tile.lastUsed = frame` と
今フレーム番号を刻む。フレームの最後に `lastUsed` が今フレームでないノードを `dispose`（GPU リソース解放）する。

```
render() {
  frame++
  …描画に使ったノードへ tile.lastUsed = frame…
  for (key, tile of cache)
    if (tile.lastUsed < frame) dispose(key, tile)   // 今フレーム未使用＝可視範囲外
}
```

これで「画面から外れた／レベルが変わって不要になったノード」が自動で片付く。LRU 的な明示的な
管理は要らず、可視判定がそのまま生存判定になる。

### 1-3. 1 フレームの新規生成数に上限（budget）を設ける

ノードの生成は `height()` を解像度ぶん（2D: 128×128、3D: 67×67）呼ぶので、まとめて大量に作ると
そのフレームだけカクッと固まる。そこで **1 フレームに作る数を制限**する。

| | 定数 | 値 | 補足 |
| --- | --- | --- | --- |
| 2D | `BUDGET_PER_FRAME` | 3 | 足りないタイルを**カメラに近い順**に N 個だけ生成 |
| 3D | `BUILD_BUDGET` | 16 | refine 用フロンティアの先行生成。生成は軽いので多め |

予算オーバーぶんは「今フレームは諦めて次フレームで作る」。その間は次の 1-4 で穴を埋める。

### 1-4. 揃うまでは粗いレベルを下敷きにする（穴を出さない）

新しいレベルのノードがまだ生成中（budget 待ち）でも、**そのフレームに必ず何かを描く**ために、
1 段以上**粗いレベルのノードをフォールバックとして描く**。粗いノードは広い範囲を 1 枚で覆える分、
ぼやけてはいても穴は空かない。子が揃ったら自然に細かい方へ切り替わる。

- 2D: 現在レベルの下に `KEEP_COARSER`(=3) 段ぶん粗いタイルを残し、**粗い→細かいの順で重ね描き**。
  読み込み中は粗いタイルが透けて見える。
- 3D: 子 4 つが「覆える」状態になるまで、**親ノード（粗い）を描画**しつつ子の生成を要求する。

## 2. 2D シーン — タイルピラミッド（Google Maps 方式）

[scene-heightmap.ts](../src/visualization/scenes/scene-heightmap.ts)。真上から見た俯瞰なので話が単純で、
**全画面で 1 つのレベルを選び**、そのレベルのタイルを敷き詰める（＝Web 地図のタイル方式）。

```
const BASE_TILE_WORLD = 256;  // レベル0タイルのワールド長（1 上がるごとに半分）
const TILE_RES        = 128;  // タイルテクスチャの一辺（全レベル共通）
const MAX_LEVEL       = 12;   // 拡大しすぎ防止
```

### 2-1. レベルの選び方：「1 テクセル ≒ 1 ピクセル」

ズーム量から、テクスチャがちょうど等倍で表示されるレベルを 1 つ選ぶ。

```
worldPerPixel = viewHeight / drawingBufferHeight      // 画面 1px が映すワールド長
texelWorld(L) = BASE_TILE_WORLD / 2^L / TILE_RES      // レベル L の 1 テクセルのワールド長
```

`texelWorld(L) ≈ worldPerPixel` になる `L` を解く:

```
BASE_TILE_WORLD / 2^L / TILE_RES = worldPerPixel
        2^L = BASE_TILE_WORLD / (TILE_RES · worldPerPixel)
          L = log2( BASE_TILE_WORLD / (TILE_RES · worldPerPixel) )
```

実コードはこれを丸めてクランプするだけ:

```ts
const level = clamp(
  Math.round(Math.log2(BASE_TILE_WORLD / (TILE_RES * worldPerPixel))),
  0, MAX_LEVEL,
);
```

ズームインで `worldPerPixel` が小さくなる → `level` が上がる → 小さいタイルに切り替わり、
`height()` を高解像度で取り直すので**いくら寄ってもくっきり**。これが「拡大コピーと違う」核心。

### 2-2. 1 フレームの流れ

```
1. ズームから level を決める（上式）
2. 可視範囲＋マージン(TILE_MARGIN)を覆う現在レベルのセル範囲 [cxMin..cxMax]×[czMin..czMax] を出す
3. そのうち未生成のタイルを集め、画面中心に近い順 (d2) にソートして BUDGET_PER_FRAME 個だけ createTile
4. 描画リスト = キャッシュ内で「可視 ∩ レベルが [level-KEEP_COARSER, level]」のタイル
5. 描画リストを 粗→細 でソートして重ね描き（粗いのが下敷き）
6. 今フレーム未使用のタイルを解放
```

タイルの中身は `buildTileData()` が作る。端を含むグリッド点（`i/(TILE_RES-1)`）で `height()` を
サンプルし `heightToColor()` で着色した RGBA。**端を共有するので隣のタイルと色がつながり継ぎ目が出ない**。

```ts
// 端を含むグリッド点で取る（i/(RES-1)）。隣接タイルが境界値を共有し継ぎ目が出ない。
const wx = ox + (i / (TILE_RES - 1)) * tileWorld;
```

### 2-3. なぜ「全画面で 1 レベル」で済むのか

真上からの平行投影に近い俯瞰では、**画面のどこでも 1px の細かさが同じ**。だから場所ごとに
レベルを変える必要がなく、全画面一律で 1 レベルを選べば足りる。これが次の 3D との決定的な違い。

## 3. 3D シーン — 適応的四分木 ＋ 視錐台カリング

[scene-heightmap3d.ts](../src/visualization/scenes/scene-heightmap3d.ts)。透視投影では**手前は大きく・奥は小さく**
映るので、「全画面で 1 レベル」では破綻する。手前は細かく、奥は粗く、と**場所ごとに深さを変える**
本物の四分木 LOD が要る。

```
const LEAF_GRID  = 64;    // 各ノードのメッシュ解像度（固定 64×64 セル, 頂点 65×65）
const ROOT_SIZE  = 4096;  // 深さ0ノードの一辺（深いほど半分）
const MAX_DEPTH  = 10;    // 最小ノード = 4096 / 2^10 = 4
const PIXEL_BUDGET = 2;   // 1 三角形の目標最大サイズ（px）。小さいほど高精細・多ノード
```

ポイントは **メッシュ解像度 `LEAF_GRID` は全ノード固定**で、詳細はノードの**世界サイズの分割**で出すこと。
深いノードほど狭い範囲を同じ 64×64 で刻むので密になる。index バッファは全ノード共通の 1 本を共有できる。

### 3-1. 分割するか？：投影三角形サイズが予算を超えたら

ノードを分割すべきかは「**このノードの 1 三角形が、画面上で何ピクセルに映るか**」で決める。
透視投影では、世界サイズ `w` の物体が距離 `d` にあると、画面上はおよそ次のピクセル数になる:

```
画面px ≈ w · screenK / d        screenK = clientHeight / (2·tan(FOV/2))
```

`screenK` は「視野の半分の高さ ÷ そのときの tan」＝ 投影の拡大率。これを 1 三角形に当てると:

```ts
const triWorld = size / LEAF_GRID;            // 1 三角形の世界サイズ
const d = nearestDist(...);                   // AABB 最近点までの距離（近い縁で十分割れる）
return (triWorld * screenK) / d > PIXEL_BUDGET; // 予算超え → 分割すべき
```

- **近い／大きいノードほど** 投影サイズが大きい → どんどん分割されて細かくなる。
- 距離はノード中心ではなく **AABB（軸並行境界箱）の最近点**で測る。縁がカメラに近いノードを
  十分に分割するため。
- `depth >= MAX_DEPTH` なら分割しない（最小サイズの底）。

### 3-2. 視錐台カリング（見えないノードは辿らない）

毎フレーム mvp 行列から視錐台の 6 平面を取り出し（`extractPlanes`）、ノードの AABB が視錐台に
一部でも入るかを判定（`aabbVisible`）。**入らないノードは再帰自体を打ち切る**。これで
「画面に映る範囲」だけに計算が集中する。

```
カメラ ●╲          視錐台（near〜far の四角錐）の中だけ
        ╲ ╲────────────────
         ╲   見える範囲     ╲      ← この中のノードだけ辿る／描く
          ╲────────────────╲
   視錐台の外のノードは AABB 判定で即カット（再帰しない）
```

判定は保守的（偽陽性あり＝本当は見えないノードを通すことはある）だが、描画結果は正しい。

### 3-3. 「子が揃うまで親を描く」を保証する仕組み

3D は場所ごとに深さが違うので、「降りる／降りない」を**穴も重なり（z-fighting）も無く**決める必要がある。
これを 2 つの再帰関数で実現している。

- **`coverable(depth,nx,nz)`**（副作用なしの判定）: そのノードを「描ける」か。
  視錐台外なら覆う必要なし＝true。生成済みなら true。末端なのに未生成なら false。
  分割すべきなら**子 4 つが全部 coverable か**を再帰で確かめる。
- **`selectRender(depth,nx,nz,out)`**（実際に集める）:
  - 視錐台外 → 何もしない。
  - 分割すべき & **子 4 つが全部 coverable** → 4 つの子へ降りる（自分は印だけ）。
  - 分割すべきだが子が未準備 → **自分（粗い）を描画リストに入れ**、子 4 つの生成を `requestBuild`
    （予算つき）して次フレームで refine。フォールバックの自分は描画対象なので予算外で必ず作る。
  - 末端 → 自分を描画リストに入れる。

```
           親ノード
          ╱  子4つ全部 coverable?
   YES ──┤                      └── NO
   降りる │                          自分（粗い）を1枚描いて、
   ┌──┬──┐                          子の生成を予約 → 次フレームで再挑戦
   │子│子│  ← それぞれ更に再帰
   ├──┼──┤
   │子│子│
   └──┴──┘
```

「**全部の子が覆えるときだけ降りる**」のがミソ。1 つでも欠けたら親 1 枚で覆う（穴が出ない）。
親と子を同時に描くことはない（重なりが出ない）。

### 3-4. スカート（skirt）— LOD 差の継ぎ目を隠す

隣り合うノードの深さが違うと、境界で頂点の数が合わず**T 字接合の隙間（クラック）**が出る。
これを各ノードの**外周を真下に垂らした壁（スカート）**で塞ぐ。

```
   深いノード（細かい）        浅いノード（粗い）
   ●─●─●─●─●─●─●─●            ●───────────────●
   │ 高さがズレると…         隙間→  ╳
   ●  ←ここに隙間（クラック）が見えてしまう

   対策：各ノードの縁を真下へ落とした壁を貼る
   ●─●─●─●
   │▓▓▓▓│ ← スカート（SKIRT_DEPTH ぶん下へ。地形全高より深いのでどんな段差も隠れる）
```

`SKIRT_DEPTH`(=32) は地形の全高（≈25.6 = 128·HEIGHT_SCALE）より深く取ってあるので、
どんなレベル差の段差でも確実に隠れる。スカートは法線・色を縁の頂点から引き継ぐので目立たない。
index バッファでは地表（64×64×2 三角形）に続けて外周（`perimeterIndices`）の壁を追加している。

### 3-5. 1 フレームの流れ

```
1. yaw/pitch/distance からカメラ位置 eye・mvp 行列を作り、視錐台 6 平面を取り出す
2. カメラ直下のルートセル ±ROOT_RADIUS(=2) の各ルートから selectRender で四分木を再帰
   → 描画ノードのリストを収集（同時に子の生成を要求）
3. 今フレーム未使用のノードを解放
4. 描画リストを drawElements（全ノード共通 index）
```

## 4. 2D と 3D の対比

| | 2D（タイルピラミッド） | 3D（適応的四分木） |
| --- | --- | --- |
| 投影 | 俯瞰（平行投影に近い） | 透視投影 |
| レベルの粒度 | **全画面で 1 レベル** | **場所ごとに深さが違う** |
| 分割基準 | 1 テクセル ≒ 1 ピクセル（ズーム量） | 投影三角形サイズ < `PIXEL_BUDGET`（距離） |
| 可視判定 | 矩形の重なり（AABB in 2D） | **視錐台カリング**（6 平面 × 3D AABB） |
| ノードの中身 | 128×128 の色テクスチャ | 65×65 の頂点メッシュ（位置/法線/高さ） |
| フォールバック | 粗いタイルを下敷きに重ね描き | 親ノードを描画 |
| 継ぎ目対策 | 端の頂点を共有（境界値一致） | スカート（縁を下に垂らす壁） |
| 基準サイズ | `BASE_TILE_WORLD = 256` | `ROOT_SIZE = 4096` |
| 最大レベル | `MAX_LEVEL = 12` | `MAX_DEPTH = 10` |
| 1F 生成上限 | `BUDGET_PER_FRAME = 3` | `BUILD_BUDGET = 16` |

要するに、**俯瞰は「1 レベル選んで敷き詰める」だけで済む**のに対し、**透視は手前と奥で必要な
細かさが連続的に変わるので、四分木を場所ごとに違う深さまで降りる**必要がある、という違い。

## 参考

  分割深さを変える「continuous / chunked LOD」の素直な実装。
- 2D の方式は Web 地図（Google Maps / OSM 等）のタイルピラミッドと同じ考え方
  （ズームレベル z、タイル座標 x/y、`2^z` で枚数が増える）。
- スカートによるクラック対策は chunked LOD 系（GeoClipmap, CDLOD など）で広く使われる手法。
